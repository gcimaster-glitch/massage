# HOGUSY 事業資料・システム統合方針書

**作成日:** 2026年3月16日
**作成者:** Manus AI プロフェッショナルチーム

## 1. はじめに

本ドキュメントは、HOGUSYの事業資料と現行システム間に存在する齟齬を解消し、両者を完全に整合させるための統合方針を定義するものです。プロフェッショナルチームによる分析の結果、主に「役割（ロール）の名称」「料金体系」「収益分配モデル」の3点において重要な差異が確認されました。本方針に基づきシステムを改修することで、事業戦略と実装の一貫性を確保します。

## 2. 委員会ディスカッション議事録（要約）

- **開催日時:** 2026年3月16日
- **参加者:** BtoBコンサルタント、BtoCコンサルタント、FinTech専門家、システムアーキテクト、UI/UXデザイナー（計5名）
- **議題:** 事業資料とシステムの齟齬の特定と、あるべき姿の定義

### 2.1. ポジティブ評価

- **BtoC担当:** 「CARE CUBE」という概念はユニークで、顧客に新しい体験価値を提供できる。Google Mapsとの連携は、即時予約のUXを飛躍的に向上させる。
- **BtoB担当:** 拠点ホスト、セラピストオフィス双方に明確な金銭的メリット（不労所得、低リスク事業拡大）を提示できており、法人営業のフックとして非常に強力。
- **FinTech専門家:** Stripe Connectによる自動報酬分配は、プラットフォームビジネスにおける資金移動の法的・経理的課題をクリアする最適解。スケーラビリティが高い。

### 2.2. ネガティブ評価（課題）

- **システムアーキテクト:** 現システムは「役割」の定義が資料と異なり、特に「セラピストオフィス」の概念が不明確。また、料金体系がハードコードされており、資料にあるような柔軟なプラン変更に対応できない。
- **UI/UXデザイナー:** 用語の不統一はユーザーの混乱を招く。「拠点ホスト」と「施設ホスト」、「セラピストオフィス」と「OFFICE」など、早急な統一が必要。
- **FinTech専門家:** 資料にある「取引手数料10%」「セラピスト40% / オフィス25% / ホスト20%」という収益分配モデルが、現在のコード（セラピスト70% / 本部30%）と全く異なる。これは事業の根幹に関わる最優先の修正事項。

## 3. 統合方針：3つの主要な修正領域

委員会での議論に基づき、以下の3つの領域を最優先で修正します。

### 3.1. 役割（ロール）定義の統一

システムのロール定義を事業資料に完全に合わせます。

| 現状のシステムロール | 資料上の役割名称 | **修正後のロール** | 役割内容 |
| :--- | :--- | :--- | :--- |
| `USER` | お客様 | `USER` | 施術を予約・利用する一般顧客 |
| `THERAPIST` | セラピスト | `THERAPIST` | 施術を提供する個人 |
| `OFFICE` | セラピストオフィス | `THERAPIST_OFFICE` | セラピストを管理・派遣する法人 |
| `HOST` | 拠点ホスト | `HOST` | CARE CUBE等の施術場所を提供する法人・個人 |
| `ADMIN` | HOGUSY本部 | `ADMIN` | システム全体を管理する運営者 |
| `AFFILIATE` | アフィリエイター | `AFFILIATE` | 紹介プログラム参加者 |

**決定事項:**
- システム内の `UserRole` 型定義を上記に修正します。
- `OFFICE` を `THERAPIST_OFFICE` に変更し、UI上の表記も「セラピストオフィス」に統一します。
- 「拠点ホスト」と「施設ホスト」の表記を「拠点ホスト」に統一します。

### 3.2. 料金体系のデータベース管理化

現在ハードコードされている月額料金や初期費用を、管理者がダッシュボードから動的に設定・変更できるよう、データベース管理に移行します。

**新設するDBテーブル:** `plans`

| カラム名 | 型 | 説明 |
| :--- | :--- | :--- |
| `id` | `string` | プランID (例: `office_virtual`, `host_cube_basic`) |
| `target_role` | `UserRole` | 対象ロール (`THERAPIST_OFFICE`, `HOST`, `THERAPIST`) |
| `plan_name` | `string` | プラン名 (例: 「バーチャル店舗プラン」) |
| `initial_fee` | `integer` | 初期費用（加盟金） |
| `monthly_fee` | `integer` | 月額費用 |
| `description` | `text` | プラン詳細説明 |
| `is_active` | `boolean` | 有効フラグ |

**決定事項:**
- `plans` テーブルをDBに作成します。
- 管理者ダッシュボードに「料金プラン管理」画面を新設し、プランのCRUD操作を可能にします。
- ユーザー登録時や契約時に、この `plans` テーブルを参照して料金を決定するロジックを実装します。

### 3.3. 収益分配モデルの修正

現在の「セラピスト70% / 本部30%」という固定配分を廃止し、資料に記載の柔軟な分配モデルを実装します。

**新設するDBテーブル:** `revenue_share_rules`

| カラム名 | 型 | 説明 |
| :--- | :--- | :--- |
| `id` | `string` | ルールID |
| `priority` | `integer` | 適用の優先順位（数値が低いほど優先） |
| `therapist_rate` | `float` | セラピストの配分率 (例: 0.40) |
| `office_rate` | `float` | セラピストオフィスの配分率 (例: 0.25) |
| `host_rate` | `float` | 拠点ホストの配分率 (例: 0.20) |
| `platform_fee_rate` | `float` | HOGUSY本部の手数料率 (例: 0.10) |
| `marketing_rate` | `float` | 販促コストの確保率 (例: 0.05) |
| `conditions` | `json` | 適用条件 (例: `{"site_type": "CARE_CUBE"}`) |
| `is_active` | `boolean` | 有効フラグ |

**決定事項:**
- `revenue_share_rules` テーブルをDBに作成します。
- 決済処理（`revenue-engine`）において、予約情報（施設タイプ、契約プラン等）に基づいてこのテーブルから最適なルールを動的に選択し、報酬を計算するロジックを実装します。
- デフォルトのルールとして、資料記載の「セラピスト40% / オフィス25% / ホスト20% / 本部10% / 販促5%」を登録します。

## 4. 次のステップ

本方針書に基づき、システムアーキテクトは具体的な「実装設計書」を作成します。設計書では、修正対象となるファイル、具体的なコードの変更点、DBスキーマのマイグレーション手順、そしてテスト計画を詳細に定義します。
